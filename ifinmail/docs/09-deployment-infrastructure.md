# Deployment & Infrastructure — Edge Cases (101–110)

## EC-101: Entrypoint Runs `chmod -R 755 /app/staticfiles` — Security Risk
**File:** `entrypoint.sh:65`
**Risk:** Medium — World-readable secrets
**Description:** The entrypoint runs `chmod -R 755 /app/staticfiles`, making all static files world-readable. If any sensitive files accidentally end up in the static directory (e.g., configuration dumps, debug output), they become accessible to any process on the system. Static files are also served by nginx, so they are world-readable on the web.
**Trigger:** Debug or config file accidentally placed in static directory.
**Fix:** Use more restrictive permissions (750) and ensure the static directory only contains files generated by `collectstatic`.

## EC-102: Nginx Template Uses `__SSL_CERT_HOSTNAME__` Placeholder Substitution Without Validation
**File:** `docker-compose.yml:378`
**Risk:** High — Nginx startup failure
**Description:** The nginx command uses `sed "s|__SSL_CERT_HOSTNAME__|${MAIL_HOSTNAME}|g"` to generate the nginx config. If `MAIL_HOSTNAME` is empty or contains sed special characters (e.g., `/`, `&`, `\`), the sed substitution can fail or produce a malformed config file, causing nginx to fail to start.
**Trigger:** `MAIL_HOSTNAME` environment variable empty or containing special characters.
**Fix:** Validate `MAIL_HOSTNAME` before the sed command; use a safer substitution method (e.g., `envsubst`).

## EC-103: No Graceful Shutdown of Gunicorn Workers
**File:** `Dockerfile:30`
**Risk:** Medium — Dropped requests during deploys
**Description:** The Gunicorn CMD line has correct timeout settings, but `stop_grace_period: 60s` in Docker Compose may not be enough for in-flight requests if the workers are busy processing. Gunicorn's `--graceful-timeout 60` gives workers 60 seconds to finish after receiving a SIGTERM, but Docker's `stop_grace_period` sends SIGKILL after 60 seconds. If the two are not synchronized, workers can be killed mid-request.
**Trigger:** Deployment while the server is processing long requests (e.g., DNS checks).
**Fix:** Ensure `stop_grace_period` in Docker Compose exceeds `--timeout + --graceful-timeout` (120 + 60 = 180s).

## EC-104: Docker Image Tags Use `:latest` — No Version Pinning
**Files:** `docker-compose.yml:84,131,167,204,233,271,306,362`
**Risk:** Medium — Uncontrolled upgrades
**Description:** All custom Docker images use `:latest` tags. This means every `docker compose pull` or rebuild deploys untested changes, potentially breaking the stack. There is no release process that creates versioned tags (e.g., `ifinmail/api:v1.2.3`).
**Trigger:** Rebuilding or pulling images without testing.
**Fix:** Implement semantic versioning for Docker images. Use git tags to drive image builds with versioned tags. Pin to specific versions in production.

## EC-105: PostgreSQL Custom.conf Placement Depends on Docker Entrypoint Ordering
**File:** `docker-compose.yml:23-25`
**Risk:** Medium — Startup race
**Description:** The Postgres command uses `echo "include_if_exists = ..." >> /etc/postgresql/postgresql.conf` before starting PostgreSQL. This appends to the config after the entrypoint has already started, but before postgres initializes. If the PostgreSQL version changes and the config path differs, the `include_if_exists` silently does nothing, and custom settings are not applied.
**Trigger:** PostgreSQL Alpine version update changes config file location.
**Fix:** Use a fixed config file mount at a known location and set `POSTGRES_CONFIG_FILE` environment variable, or validate the config path exists.

## EC-106: TLS Certificates Auto-Setup for PostgreSQL May Fail Silently
**File:** `docker-compose.yml:26-35`
**Risk:** Medium — PostgreSQL without TLS
**Description:** The PostgreSQL container copies TLS certs from Let's Encrypt with a `if [ -d "/etc/letsencrypt/live/${MAIL_HOSTNAME}" ]` check. If `MAIL_HOSTNAME` is empty or the directory doesn't exist during first startup (certbot hasn't run yet), SSL is silently not configured for PostgreSQL. The script returns success even though TLS was not enabled.
**Trigger:** First-time startup before certbot has issued certificates.
**Fix:** Check the exit status of the TLS setup section and log a clear warning when PostgreSQL TLS could not be configured.

## EC-107: No Mutex on `provision.sh` — Concurrent Provisioning Corrupts State
**File:** `provision.sh`
**Risk:** Medium — Configuration corruption
**Description:** `provision.sh` modifies `.env`, generates TLS certs, DKIM keys, GPG keys, builds Docker images, and starts services. If run concurrently (two terminal sessions), the scripts race on file writes, leading to a corrupted `.env` file, overlapping key generations, or half-started services.
**Trigger:** Two admins run `make provision` simultaneously.
**Fix:** Add a PID file lock at the start of `provision.sh` that prevents concurrent execution.

## EC-108: Docker Compose Healthcheck Depends on Unset Variable
**File:** `docker-compose.yml:355`
**Risk:** Medium — Health check failure
**Description:** The API container's healthcheck uses `host=os.environ['ALLOWED_HOSTS'].split(',')[0].strip()` inside a Python one-liner. If `ALLOWED_HOSTS` is empty or not set (which causes a crash in `production.py` but not in the healthcheck), the healthcheck itself raises a `KeyError`, which is caught by Docker and reported as an unhealthy container, even though the application is actually running.
**Trigger:** Startup sequence where ALLOWED_HOSTS is not yet populated.
**Fix:** Use `os.environ.get('ALLOWED_HOSTS', 'localhost').split(',')[0].strip()` or fall back to `localhost`.

## EC-109: Nginx Resolver for OCSP Stapling Uses Public DNS — Not Needed
**File:** `nginx/conf.d/ifinmail.conf:46`
**Risk:** Low — Unnecessary external dependency
**Description:** The nginx configuration specifies `resolver 1.1.1.1 8.8.8.8 valid=300s` for OCSP stapling, which is disabled (`ssl_stapling off`). The resolver is configured but never used. This is dead configuration that adds confusion and an unnecessary dependency on external DNS for no benefit.
**Trigger:** None (dead code).
**Fix:** Remove the `resolver` and `resolver_timeout` directives since OCSP stapling is disabled. Enable it properly or remove it entirely.

## EC-110: Firewall Script Assumes UFW But Doesn't Check Availability
**File:** `provisioning/scripts/firewall.sh`
**Risk:** Medium — No firewall
**Description:** The firewall script uses `ufw` commands. If UFW is not installed or not enabled, the script runs commands that may fail or silently do nothing. On a VPS without UFW, no firewall rules are applied, leaving all ports exposed (including port 25, 143, 993, 465, 587).
**Trigger:** Running provision.sh on a system without UFW enabled.
**Fix:** Check for UFW availability at the beginning of the firewall script; provide alternative iptables rules for systems without UFW.
